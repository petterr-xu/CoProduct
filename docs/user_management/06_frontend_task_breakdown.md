# 前端任务拆解文档 - user_management
> Version: v0.5.0
> Last Updated: 2026-03-12
> Status: Draft

## 1. Phase 1 任务

目标：完成登录与业务鉴权切换，保证既有预审流程可继续使用。

1. FE-1.1 新增认证类型与枚举
- 定义 `Role/UserStatus` 与认证响应类型。
- 更新 `src/types` 与 schema 校验。
2. FE-1.2 实现认证 API Client
- 新增 `key-login/refresh/logout/me` 请求封装。
- 统一错误码解析。
> Obsolete in v0.2.0: FE-1.2 中“refresh body 传 token”实现路径不再使用。
3. FE-1.2a 刷新/登出 Cookie 化改造（v0.2.0）
- `refresh/logout` 改为 Cookie + `X-CSRF-Token` 方案。
- 删除前端持久化 refresh token 的实现。
4. FE-1.3 接入认证状态管理
- 新建 `auth-store`，管理 token 与当前用户。
- 启动时执行 `me` 校验。
5. FE-1.4 登录页与路由守卫
- 新增 `/login` 页面。
- 为业务页面增加 route guard。
6. FE-1.5 业务请求鉴权改造
- `api-client` 注入动态 `Authorization`。
- 401 场景触发 refresh 并重试一次。
7. FE-1.6 退出登录与异常处理
- `logout` 清理本地状态。
- `USER_DISABLED/PERMISSION_DENIED` UI 提示。
8. FE-1.7 数据可见范围联调（v0.2.0）
- `MEMBER` 仅本人数据、`VIEWER` 只读的前端行为校验。
- 无权限操作按钮禁用并显示原因。

## 2. Phase 2 任务

目标：提供团队管理最小可用前端能力。

1. FE-2.1 管理入口与权限门禁
- 管理入口仅 `OWNER/ADMIN` 可见。
2. FE-2.2 用户管理页面（简版）
- 用户列表、状态筛选、创建用户。
3. FE-2.3 角色与状态操作
- 修改用户状态与角色。
4. FE-2.4 API Key 管理页面（简版）
- 签发 key（一次性展示明文）。
- 吊销 key。
5. FE-2.5 审计日志查询页面（只读）
- 支持基础筛选与分页。
6. FE-2.6 管理列表契约对齐（v0.2.0）
- 用户/API Key 列表统一 `items/total/page/pageSize`。
- 列表筛选参数与后端契约一致。

## 3. Phase 3..N 任务

目标：安全增强与企业化扩展。

1. FE-3.1 邀请制注册与激活 UI。
2. FE-3.2 个人会话管理页面（查看与踢出会话）。
3. FE-3.3 安全中心（异常提示、风控反馈）。

## 3A. Phase 4 任务（v0.3.0 新增）

目标：完成用户/成员建模细化、权限治理红线前端承接、职能角色能力接入。

1. FE-4.1 成员模型视图升级
- 将管理页视图从“用户”语义升级为“成员”语义。
- 前端类型拆分 `UserAccountView` 与 `MembershipView`。
2. FE-4.2 权限角色可读化改造
- 枚举值统一映射中文标签，避免直接展示原始枚举。
- 管理页、顶栏、详情页统一采用标签映射。
3. FE-4.3 高风险操作门禁提示
- 对自降权/自禁用/修改 owner 等场景显示禁用状态与原因提示。
- 与后端拒绝错误码做统一映射文案。
4. FE-4.4 职能角色管理页
- 新增职能角色列表/新增/启停用页面。
- 支持按职能角色筛选成员。
5. FE-4.5 成员职能单值绑定
- 成员编辑仅允许选择一个职能角色（单选）。
- 提交前校验 `functionalRoleId` 合法性。
6. FE-4.6 契约兼容与迁移收口
- 兼容 `/api/admin/users/*` 与 `/api/admin/members/*` 过渡期差异。
- 完成路径切换后下线旧调用路径。

## 3B. Phase 5 任务（v0.4.0 新增）

目标：以登录上下文驱动组织选择，消除创建成员的组织歧义，并为多组织登录预留前端能力。

1. FE-5.1 接入登录上下文接口（关联 TD-401 / FC-401）
- 新增 `GET /api/auth/context` API client 与类型定义。
- 在管理页初始化流程中加载并缓存 `activeOrg/availableOrgs/scopeMode`。
2. FE-5.2 创建成员组织输入改造（关联 TD-402 / FC-402）
- 移除 `orgId` 文本输入框。
- 改为组织下拉组件，数据源仅来自 `availableOrgs`。
3. FE-5.3 ORG_SCOPED 行为固化（关联 TD-403 / FC-402）
- 当 `scopeMode=ORG_SCOPED` 时默认选中 `activeOrg` 并锁定不可编辑。
- 表单提交时组织参数仅允许使用当前上下文组织。
4. FE-5.4 无组织引导与错误处理（关联 TD-403 / FC-403）
- 当 `activeOrg=null` 时禁用创建按钮并展示引导态。
- 映射 `NO_ACTIVE_ORG` 错误码到统一提示。
5. FE-5.5 未来 USER_SCOPED 交互预埋（关联 TD-404 / FC-401）
- 保留下拉多组织渲染能力，不在本阶段开放切换入口。
- 在 UI 与状态管理中保留 `scopeMode` 分支。
6. FE-5.6 回归测试补齐（关联 TD-401~TD-404）
- 覆盖 context 加载成功/失败、组织下拉受控、无组织禁用与错误码提示。

## 3C. Phase 6 任务（v0.5.0 新增）

目标：提升 API Key 签发可用性，收敛目标成员选择语义，并为多组织签发预留交互能力。

1. FE-6.1 API Key 签发组织选择改造（关联 TD-503/TD-505 / FC-502）
- 在签发表单接入组织上下文：`ORG_SCOPED` 锁定组织，`USER_SCOPED` 预留可切换分支。
- 组织变更时清空成员候选与已选目标，避免跨组织误提交。
2. FE-6.2 成员联想搜索组件（关联 TD-501/TD-502 / FC-501）
- 接入 `GET /api/admin/member-options`，按邮箱/显示名前缀搜索。
- 增加防抖、最小字符长度、请求取消，降低高频请求抖动。
3. FE-6.3 替换手工 userId 输入（关联 TD-501 / FC-501）
- 移除签发表单中 `userId` 纯文本主输入。
- 仅允许从候选项选择后提交 `userId`。
4. FE-6.4 API Key 列表可读字段接入（关联 TD-504 / FC-503）
- 列表展示 `userDisplayName/userEmail`，保留 `userId` 作为次要信息。
5. FE-6.5 错误处理与空态增强（关联 TD-503 / FC-502）
- 对 `NO_ACTIVE_ORG/PERMISSION_DENIED/RESOURCE_NOT_FOUND` 提供清晰提示。
- 无候选成员时提供可操作引导（调整组织/关键词）。
6. FE-6.6 回归测试补齐（关联 TD-501~TD-505）
- 覆盖组织切换、联想选择、空候选、签发失败与成功展示。

## 4. 依赖与执行顺序

### 4.1 依赖

1. FE-1.2 依赖后端认证接口可用。
2. FE-1.4 依赖 FE-1.3 状态管理完成。
3. FE-1.5 依赖 FE-1.2、FE-1.3 完成。
4. FE-2.x 依赖后端管理 API 与权限模型稳定。
5. FE-4.x 依赖后端成员模型与职能角色接口稳定。
6. FE-5.x 依赖 BE-5.1（`auth/context`）与 BE-5.2（创建成员组织约束）完成。
7. FE-6.x 依赖 BE-6.1（成员检索接口）与 BE-6.2/BE-6.3（签发组织语义）完成。

### 4.2 推荐顺序

1. 类型定义 -> API Client -> 状态管理 -> 登录页/守卫 -> 请求注入改造 -> 异常收口。
2. 管理页面放在 Phase 2，避免阻塞鉴权主链路。
3. Phase 4 先改类型与枚举映射，再改管理交互，最后切换成员语义接口。
4. Phase 5 先接 `auth/context`，再替换创建成员组织输入组件，最后补齐无组织与未来模式分支。
5. Phase 6 先落成员检索 API 封装，再改签发表单交互，最后升级列表可读字段与回归用例。

## 5. 风险与缓解

1. 风险：token 刷新逻辑导致重复请求或死循环。
- 缓解：全局 refresh 锁 + 单次重试上限。
2. 风险：角色门禁与后端不一致。
- 缓解：以前端“展示控制”为辅，后端“权限判定”为准。
3. 风险：旧页面遗留静态 token 逻辑。
- 缓解：全局检索 `NEXT_PUBLIC_API_TOKEN` 并替换，纳入回归清单。
4. 风险：治理红线收紧后，前端仍展示可操作按钮导致误导。
- 缓解：补充前端按钮禁用规则与明确错误提示，且以后端返回为最终准则。
5. 风险：上下文接口失败导致管理页无法完成写操作。
- 缓解：创建操作强依赖 context；失败时降级只读并提供重试。
6. 风险：成员联想组件与签发提交状态不同步导致误提交。
- 缓解：提交前校验“已选成员对象”而非输入文本；组织切换强制清空已选值。

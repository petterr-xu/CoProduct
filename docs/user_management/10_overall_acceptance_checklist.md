# 总体检查清单 - user_management
> Version: v0.4.0
> Last Updated: 2026-03-12
> Status: Draft

## 1. 端到端链路检查

1. 管理员签发 API Key -> 用户登录 -> 发起预审 -> 查询历史，链路完整可用。
2. 用户 token 过期后自动刷新，业务流程不中断。
3. 用户被禁用后，已有会话失效并无法继续访问业务接口。
4. 吊销 API Key 后无法再次登录。
5. `MEMBER` 无法访问他人业务数据，`VIEWER` 无法执行写接口。
6. 管理端成员治理链路（角色调整/状态调整/职能调整）执行与拒绝路径均符合预期。
7. AC-E2E-501：登录后前端可拿到统一上下文并展示当前组织。
8. AC-E2E-502：创建成员时组织来源受控（下拉），不可手工输入任意 orgId。
9. AC-E2E-503：无组织上下文场景下，前后端协同提示并阻断写操作。

## 2. 前后端契约兼容性检查

1. `04_frontend_contract.md` 与 `05_backend_contract.md` 路径、方法、字段完全一致。
2. 前端错误码分支覆盖后端所有认证域错误码。
3. 业务接口在鉴权升级后不发生 payload 破坏性变更。
4. 枚举值域（角色、状态）前后端一致且无大小写偏差。
5. refresh/logout 采用一致的 Cookie + CSRF 契约，无 body token 歧义。
6. 成员模型字段（权限角色/职能角色/成员状态）在前后端契约中语义一致。
7. `GET /api/auth/context` 在 `04/05` 中字段、枚举、错误场景保持一致。

## 3. 阶段完成度检查（Phase 1..N）

| 阶段 | 前端完成标准 | 后端完成标准 | 结果 |
|---|---|---|---|
| Phase 1 | 登录、守卫、token 注入、401 refresh、Cookie+CSRF | auth API、JWT、业务鉴权切换、数据归属、权限矩阵落地 | ☐ |
| Phase 2 | 管理页最小功能可用 | 管理员 API、权限策略、审计查询 | ☐ |
| Phase 3 | 邀请激活、安全中心与会话页 | 邀请流程、限流风控、OIDC 预留 | ☐ |
| Phase 4 | 成员模型 UI、职能角色管理、治理红线交互提示 | 身份模型细化、治理红线约束、职能角色接口与迁移 | ☐ |
| Phase 5 | 上下文驱动组织选择、无组织引导、未来多组织交互预埋 | `auth/context`、创建成员组织约束收敛、`NO_ACTIVE_ORG` 与审计 | ☐ |

## 4. 发布 Gate

### 4.1 必须满足

1. Phase 1 全量通过（前后端 + 联调 + 回归）。
2. 核心鉴权链路有自动化测试（至少登录、刷新、登出、鉴权失败）。
3. 生产配置中禁用静态开发 token 兜底。
4. 敏感数据（API Key 明文）不落库、不落日志。

### 4.2 建议满足

1. Phase 2 的管理 API 与最小管理页面一并交付。
2. 审计日志支持导出并用于安全复盘。
3. 发布前完成一次权限穿透测试（跨用户数据访问应被拒绝）。
4. 发布前完成“至少一个 owner”与“危险自操作拦截”专项回归。

# 系统现状评估与下一期开发方向
> Version: v0.1.0
> Last Updated: 2026-03-12
> Status: Active

## 1. 现状评估（MVP 视角）

## 1.1 已达成目标

1. 形成“输入 -> 编排 -> 结构化输出 -> 历史回看”的完整闭环。
2. 关键接口契约稳定，前后端联调路径可复现。
3. 支持版本链 regenerate、附件上传、历史筛选分页。
4. 关键节点有基础降级机制，系统可在局部失败下继续输出。

## 1.2 主要短板

1. AI 能力仍是 Heuristic，本质是规则模拟，不是生产级智能推理。
2. RAG 检索精度与规模能力有限，语料治理能力较弱。
3. 流程可观测性不足（缺少统一 trace id、指标面板、告警）。
4. 数据治理与迁移能力不足（`create_all` 模式、无 Alembic）。
5. 自动化测试覆盖不均，缺少 API 集成测试与前端 E2E。

---

## 2. 可改进点（按优先级）

## 2.1 P0：质量与稳定性

1. 接入云模型客户端（保持 ModelClient 协议）并引入降级回退链。
2. 为 workflow 增加节点级超时、重试、熔断策略。
3. 引入统一请求追踪：`request_id/session_id/node_name` 全链路一致。
4. 统一错误码体系与异常映射（补齐 `PERSISTENCE_ERROR` 等边界）。

## 2.2 P1：RAG 与知识治理

1. 向量检索升级（pgvector 或专用向量库）。
2. 文档接入链路：解析、切片、清洗、去重、版本化。
3. 证据质量评分：相关性 + 可信度 + 新鲜度多因子融合。
4. 增加检索评测集与回归基线。

## 2.3 P1：数据与工程化

1. 引入 Alembic 管理 schema 演进。
2. 关键表增加索引策略（history 查询、session/version 查找、evidence 回溯）。
3. 附件解析异步化（任务队列）并支持更多格式（pdf/docx）。
4. 将本地文件存储升级为对象存储（S3 兼容）。

## 2.4 P2：产品体验

1. 版本链可视化（父子 session 关系图）。
2. 结果页展示“证据引用与结论映射关系”。
3. 前端增加解析进度提示与可恢复操作（重试、忽略、替换）。
4. 历史页增加排序项、组合筛选和快速比较视图。

---

## 3. 下一期建议目标（建议命名：M4）

## 3.1 M4 总体目标

从“可用 MVP”升级到“可验证质量 + 可运维 + 可扩展”的 Beta 版本。

## 3.2 M4 分阶段建议

### 阶段 A（2~3 周）：底层能力升级

1. 云模型接入 + Heuristic 兜底回退。
2. RAG 检索能力升级（至少实现真正向量检索）。
3. 引入统一链路追踪和基础指标。

### 阶段 B（2~3 周）：质量与治理

1. 建立评测集（能力判断正确率、证据覆盖率）。
2. 加入 API 集成测试与前端关键 E2E。
3. 数据迁移与索引优化。

### 阶段 C（1~2 周）：体验收口

1. 版本链与证据映射可视化。
2. 文件解析异步状态体验优化。
3. 联调手册升级为发布 checklist。

---

## 4. 建议的交付指标（可量化）

1. 预审成功率（非 FAILED）>= 95%。
2. 空证据率下降（目标 < 10%，按样本集）。
3. 历史查询 P95 < 300ms（本地环境外需实测）。
4. 前端关键流程（创建/详情/regenerate）E2E 用例通过率 100%。
5. 节点异常可观测率 100%（有日志、有错误码、有上下文）。

---

## 5. 风险提醒

1. 若直接接云模型但不做 schema/guardrail，输出漂移会放大。
2. 若先做 UI 扩展但不先提升 RAG 质量，用户会更快暴露结果可信度问题。
3. 若不补迁移和监控，功能增长后定位和回滚成本会迅速上升。
